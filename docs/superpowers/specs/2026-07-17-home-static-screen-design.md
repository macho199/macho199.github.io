# Developer Blog 홈 정적 화면 설계

## 목적

현재 Gatsby 홈을 디자인 참조 `index.html`의 정적 목록 화면으로 교체한다. 실제 MDX 게시글만 표시하고, 아직 구현하지 않은 필터·페이지네이션 같은 컨트롤은 노출하지 않는다.

## 범위

### 포함

- 숨김 페이지 제목과 한국어 블로그 소개 문구
- 실제 MDX 게시글의 발행일 역순 목록
- 게시글 태그, 제목 링크, 설명, 발행일
- 게시글이 없을 때의 빈 상태
- 1440px·1020px·390px 반응형 표현

### 제외

- 게시글 개수
- 태그 필터와 태그 더보기
- 페이지네이션
- 읽는 시간 계산과 표시
- 테스트용 가짜 게시글
- 포스트 상세 화면 변경

제외한 기능은 각각의 후속 구현 단계에서 데이터와 동작을 함께 추가한다.

## 컴포넌트 경계

### 홈 페이지

- GraphQL로 MDX 게시글을 발행일 역순 조회한다.
- GraphQL 결과를 표현 컴포넌트가 사용할 타입으로 전달한다.
- 페이지 데이터 조회와 SEO 경계만 소유하고 카드의 표현 책임을 갖지 않는다.

### `HomeIntro`

- 문서의 유일한 `h1`인 `개발자 블로그`를 시각적으로 숨겨 제공한다.
- 다음 소개 문구를 화면에 표시한다.

  > AI를 활용한 개발 워크플로우, 백엔드 구현 기록, 제품 관점의 엔지니어링 판단을 짧고 실용적으로 정리하는 블로그입니다.

### `PostList`

- 번호 표시는 없지만 게시 순서에 의미가 있는 `ol`을 사용한다.
- 보조기술용 숨김 제목으로 목록 영역을 설명한다.
- 게시글 배열이 비어 있으면 `아직 게시글이 없습니다.`를 표시한다.
- 각 게시글의 표현은 `PostCard`에 위임한다.

### `PostCard`

- 태그, 제목, 설명, 발행일 순서로 표시한다.
- 제목은 `/posts/<slug>/`로 이동하는 실제 Gatsby 링크다.
- 태그는 현재 단계에서 상호작용 없는 정보로만 표시한다.
- 화면 날짜는 `YYYY.MM.DD`, `time`의 `dateTime`은 원본 날짜를 사용한다.

## 데이터 흐름

```text
MDX frontmatter
  -> Gatsby 홈 GraphQL 쿼리(발행일 내림차순)
  -> 홈 페이지의 타입 지정 데이터
  -> PostList
  -> PostCard
```

- 기존 필수 frontmatter인 `title`, `slug`, `publishedAt`, `description`, `tags`를 사용한다.
- GraphQL 쿼리는 날짜 원본과 화면 표시값을 별도 필드로 제공한다.
- 표현 컴포넌트는 GraphQL에 직접 의존하지 않는다.
- 기존 `Seo`와 Gatsby Head API는 변경하지 않는다.

## 시각·반응형 규칙

- 기존 920px 공통 콘텐츠 컨테이너와 디자인 토큰을 재사용한다.
- 소개 영역은 위 32px, 아래 16px 여백을 둔다.
- 목록 영역은 위 16px, 아래 32px 여백을 둔다.
- 목록에는 위쪽 구분선을, 각 카드에는 아래쪽 구분선을 둔다.
- 카드는 세로 24px 여백을 사용하고, 내부 콘텐츠 폭은 최대 760px다.
- 카드 내부 간격은 태그·제목·설명·날짜 사이에 일관되게 적용한다.
- 제목만 링크로 동작하며 카드 배경은 포인터를 올렸을 때 은은하게 변한다.
- 태그와 긴 텍스트는 좁은 화면에서 자연스럽게 줄바꿈한다.
- 1440px·1020px·390px에서 페이지 가로 넘침이 없어야 한다.

## 접근성

- 페이지에는 숨김 `h1` 하나를 유지한다.
- 목록 영역에는 보조기술용 숨김 제목을 둔다.
- 게시글 목록은 `ol`, 각 항목은 `li`와 `article`로 표현한다.
- 발행일은 `time` 요소로 표현한다.
- 상호작용하지 않는 태그를 버튼이나 링크처럼 보이게 만들지 않는다.
- 카드 전체를 클릭 영역으로 오인시키지 않고 제목 링크에만 명확한 포커스 표현을 제공한다.

## 오류·빈 상태

- 필수 MDX 데이터 오류는 기존 스키마와 빌드 검증에 맡기고 임의 기본값으로 숨기지 않는다.
- 게시글이 0개인 정상 상태는 목록 대신 간단한 안내 문구로 처리한다.
- 동작하지 않는 필터·페이지네이션 컨트롤은 비활성 상태로 미리 렌더링하지 않는다.

## 검증과 완료 조건

- `HomeIntro`, `PostList`, `PostCard`의 책임과 페이지 데이터 경계를 테스트한다.
- 실제 게시글 링크, 태그, 설명, 화면 날짜와 원본 `dateTime`이 생성 결과에 포함되는지 확인한다.
- 게시글이 없을 때 빈 상태가 렌더링되는 경로를 테스트한다.
- 숨김 제목과 의미 있는 목록 구조를 확인한다.
- 게시글 개수, 필터, 페이지네이션, 읽는 시간이 포함되지 않았는지 확인한다.
- `npm test`가 통과한다.
- `npm run typecheck`가 통과한다.
- `npm run clean` 이후 `npm run build`가 통과한다.
- 기존 스타일·공통 레이아웃 검증이 계속 통과한다.
- 1440px·1020px·390px에서 줄바꿈, 가로 넘침, 주요 간격을 확인한다.

픽셀 단위 스냅샷이나 CSS 문자열 전체를 고정하는 테스트는 만들지 않는다.
