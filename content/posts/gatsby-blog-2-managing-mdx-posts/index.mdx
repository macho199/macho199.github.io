첫 번째 글에서는 배포 산출물만 남은 블로그를 Gatsby 5 소스로 다시 구성하고, MDX와 스타일 기반을 분리해 검증한 과정을 정리했습니다. 소스가 다시 생겼다고 해서 곧바로 글을 안정적으로 추가할 수 있는 것은 아니었습니다.

글마다 파일 위치와 메타데이터 형식이 달라지거나 잘못된 날짜와 중복 URL이 빌드 후반까지 숨어 있으면, 글이 늘어날수록 관리 비용이 커집니다. 이번 단계에서는 MDX 파일 하나를 단순 문서가 아니라 빌드가 검사하는 콘텐츠 계약으로 다뤘습니다.

## 포스트 파일과 메타데이터를 한 단위로 묶기

각 글은 다음 구조를 사용합니다.

```text
content/posts/<slug>/index.mdx
```

현재 글은 `content/posts/gatsby-blog-2-managing-mdx-posts/index.mdx`에 있습니다. 디렉터리 이름과 공개 URL에 쓰는 slug를 맞추면 저장소에서 글을 찾기 쉽고, 나중에 이미지가 필요할 때 같은 디렉터리 안에 둘 수 있습니다.

MDX 파일의 맨 위에는 YAML frontmatter를 둡니다.

```yaml
title: "Gatsby로 블로그 만들기 2편 - MDX 포스트 관리"
slug: "gatsby-blog-2-managing-mdx-posts"
publishedAt: "2026-05-16"
description: "MDX 글을 디렉터리 단위로 관리하고 frontmatter 스키마와 빌드 전 검증으로 잘못된 날짜·slug·태그를 차단한 과정을 정리합니다."
tags:
  - Gatsby
  - MDX
  - GraphQL
  - Validation
```

각 필드는 여러 화면에서 함께 사용됩니다.

| 필드 | 역할 |
| --- | --- |
| `title` | 홈 카드, 상세 제목, 문서 title |
| `slug` | `/posts/<slug>/` 공개 경로 |
| `publishedAt` | 홈 정렬, 화면 날짜, article metadata |
| `description` | 홈 요약, 상세 설명, meta description |
| `tags` | 홈·상세 태그, article tag metadata |

같은 값을 컴포넌트나 SEO 설정에 다시 적지 않고 frontmatter를 단일 원본으로 사용합니다.

## 파일을 Gatsby 데이터 레이어에 연결하기

`gatsby-source-filesystem`은 `content/posts` 아래 파일을 `posts`라는 소스로 읽습니다.

```javascript
{
  resolve: "gatsby-source-filesystem",
  options: {
    name: "posts",
    path: path.join(rootDirectory, "content", "posts"),
  },
}
```

이 플러그인이 파일을 `File` 노드로 만들고, `gatsby-plugin-mdx`가 MDX 내용을 `Mdx` 노드로 변환합니다. 파일을 찾는 책임과 MDX를 해석하는 책임을 분리했기 때문에 다른 파일 소스가 추가되더라도 `sourceInstanceName`으로 블로그 글의 경계를 확인할 수 있습니다.

MDX 본문에는 표와 체크 목록 같은 GitHub Flavored Markdown을 사용할 수 있도록 `remark-gfm`만 연결했습니다. 글 두 편을 추가하기 위해 새 플러그인이나 런타임 의존성을 늘리지는 않았습니다.

## frontmatter를 명시적 계약으로 고정하기

Gatsby는 데이터에서 GraphQL 스키마를 추론할 수 있습니다. 하지만 글이 적을 때 우연히 모든 값이 채워져 있거나 특정 배열이 비어 있으면 추론 결과가 콘텐츠 상태에 따라 달라질 수 있습니다.

그래서 `createSchemaCustomization`에서 frontmatter 타입을 직접 정의했습니다.

```graphql
type MdxFrontmatter @dontInfer {
  title: String!
  slug: String!
  publishedAt: Date! @dateformat
  description: String!
  tags: [String!]!
}

type Mdx implements Node {
  frontmatter: MdxFrontmatter!
}
```

`@dontInfer`는 실제 글의 값으로 타입을 추측하지 않고 선언한 필드만 계약으로 사용하게 합니다. 느낌표가 붙은 필드는 빠질 수 없고, `publishedAt`은 Gatsby의 날짜 형식 기능을 사용할 수 있습니다.

이 스키마는 GraphQL의 형태를 고정합니다. 다만 `2026-02-31`처럼 문자열 모양은 날짜와 비슷하지만 실제 달력에는 없는 값이나, 서로 다른 글이 같은 slug를 쓰는 문제까지 모두 설명하지는 못합니다. 작성자 입력 규칙은 별도 검증기로 다룹니다.

## 페이지 생성 전에 작성자 입력 검증하기

`validatePostNodes`는 Gatsby가 정적 라우트를 만들기 전에 전체 글의 frontmatter를 검사합니다.

```javascript
const requiredStringFields = ["title", "slug", "publishedAt", "description"]

for (const field of requiredStringFields) {
  const value = frontmatter[field]

  if (typeof value !== "string" || value.trim() === "") {
    errors.push(`Post ${post.id} requires a non-empty ${field}.`)
  }
}
```

태그는 배열이어야 하고 빈 문자열을 포함할 수 없습니다. slug는 소문자 영문·숫자와 단일 하이픈만 허용합니다.

```javascript
const slugPattern = /^[a-z0-9]+(?:-[a-z0-9]+)*$/
```

검증기는 전체 글을 한 번 순회하면서 slug 소유자를 기억합니다. 같은 slug가 다시 나오면 두 글의 식별자를 함께 오류로 남깁니다.

```text
Duplicate post slug "same-slug" in first-post and second-post.
```

`publishedAt`은 정규식으로 `YYYY-MM-DD` 모양을 먼저 확인한 뒤 월별 일수와 윤년 규칙을 계산합니다. 그래서 `2024-02-29`는 허용하지만 `2026-02-29`, `2026-02-31`, `2026-13-01`은 거부합니다.

JavaScript의 느슨한 날짜 파싱에 맡기지 않고 작성 형식과 실제 달력 날짜를 함께 고정한 이유는 홈 정렬과 SEO 발행일이 같은 값을 사용하기 때문입니다.

## 오류를 모아 빌드를 중단하기

첫 오류에서 바로 예외를 던지지 않고 모든 글을 검사해 오류 배열을 만듭니다. 한 번의 빌드에서 누락된 title, 잘못된 날짜, 중복 slug를 함께 볼 수 있습니다.

```javascript
const validationErrors = validatePostNodes(posts)

if (validationErrors.length > 0) {
  reporter.panicOnBuild(
    `Invalid MDX post metadata:\n${validationErrors.join("\n")}`,
  )
  return
}
```

GraphQL 자체가 실패하거나 데이터가 없을 때도 `reporter.panicOnBuild`로 중단합니다. 잘못된 글을 제외하고 나머지만 조용히 배포하면 작성자는 누락을 발견하기 어렵기 때문입니다.

검증 책임은 세 층으로 나눴습니다.

1. GraphQL 스키마는 필드의 타입과 필수 여부를 고정합니다.
2. `validatePostNodes`는 작성자 입력과 글 사이의 규칙을 검사합니다.
3. 프로덕션 검증기는 실제 URL·HTML·SEO·sitemap 산출물을 확인합니다.

## 새 글을 추가하는 실제 순서

새 글을 추가할 때는 다음 순서를 사용합니다.

1. `content/posts/<slug>/index.mdx`를 만듭니다.
2. 다섯 frontmatter 필드를 채웁니다.
3. 본문과 필요한 로컬 자산을 같은 글 경계에서 관리합니다.
4. 소스 계약 테스트와 타입 검사를 실행합니다.
5. 깨끗한 Gatsby 빌드에서 URL을 생성합니다.
6. 홈 목록, 상세 페이지, SEO와 sitemap을 산출물에서 확인합니다.

```shell
npm test
npm run typecheck
npm run build
npm run verify:home
npm run verify:post
```

이 흐름에서는 글 작성 실수가 배포 후 404나 잘못된 메타데이터로 나타나기 전에 빌드에서 실패합니다. 콘텐츠 수가 늘어나도 새 글이 따라야 할 계약은 동일합니다.

## 마치며

MDX를 선택한 이유는 Markdown 안에 React 컴포넌트를 쓸 수 있어서만은 아니었습니다. 파일, 메타데이터, 검증, 공개 URL을 하나의 재현 가능한 흐름으로 관리할 수 있다는 점이 더 중요했습니다.

다음 글에서는 유효한 MDX 노드가 홈의 발행일 역순 목록과 `/posts/<slug>/` 상세 페이지로 나뉘어 생성되는 GraphQL 흐름을 정리합니다.
