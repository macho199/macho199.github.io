승인된 읽기 화면을 Gatsby 템플릿에 적용하고, 기존 공개 글 한 편과 이미지 두 장을 실제 콘텐츠로 이전하는 구현 계획입니다.

한 줄 요약

타입이 지정된 포스트 헤더, 범위가 제한된 본문 CSS, 보수적으로 교정한 실제 글, 프로덕션 검증을 다섯 단계로 구현합니다.

01배경 · 왜 지금 하는가

홈 정적 화면은 완성됐지만 포스트는 아직 기반 확인용 샘플과 최소 마크업만 사용하고 있습니다.

이번 작업의 목적은 UI 모양만 바꾸는 것이 아닙니다. 실제 글 하나를 기준으로 템플릿, 콘텐츠, 자산, SEO와 빌드 검증이 같은 경로를 바라보게 만드는 것입니다.

작업 관리

GitHub #18이 현재 구현 단위이며, 전체 UI 재구축 진행 상태는 Redmine #26이 관리합니다.

02접근 · 설계 결정

데이터 조회, 메타데이터 표현, 본문 스타일을 서로 다른 경계에 둡니다.

①MDX frontmatter
②MDX body
→
post.tsxGraphQL · 페이지 조립
→
화면PostHeader · post.css

템플릿은 조회와 조립만 맡고, 표현 컴포넌트와 CSS는 GraphQL을 알지 않습니다.

핵심 결정

결정이유
의미 있는 MDX + 범위 제한 CSS본문에 Tailwind 클래스를 섞지 않고 글과 외형을 분리합니다.
PostHeader만 분리메타데이터 표현은 재사용하되 단순 전달용 PostBody는 만들지 않습니다.
실제 글로 샘플 교체홈과 포스트 검증이 제품에서 사용할 데이터로 동작하게 합니다.
이미지 자체 제공예전 사이트 URL을 런타임 의존성으로 남기지 않습니다.

선택하지 않은 대안

대안이번에 선택하지 않은 이유
MDX 컴포넌트 매핑현재 글 한 편의 기본 의미 요소에는 추가 추상화가 이릅니다.
본문에 Tailwind 클래스 직접 작성콘텐츠가 특정 화면 구현에 묶이고 교정하기 어려워집니다.
이미지 플러그인 도입이미지 두 장을 옮기는 범위보다 의존성과 설정이 커집니다.
예전 URL 리다이렉트종성님이 이번 출시에서는 새 URL만 사용하기로 확정했습니다.

03확정 vs 추정

확정 새 경로, 원본 발행일, 태그 네 개, 이미지 두 장, 제외 기능과 컴포넌트 경계는 종성님이 승인했습니다.

확정 현재 Gatsby 공식 빠른 시작과 Tailwind PostCSS 문서를 다시 확인했고, 글에는 npm init gatsby와 npm run serve를 사용합니다.

추정 1440px·1020px·390px에서 세부 여백은 초기 CSS만으로 충분할 가능성이 높습니다. 근거는 승인된 HTML 산출물과 기존 920px 컨테이너이며, 최종 값은 실제 브라우저 검증 후에만 확정합니다.

실행 중 확인 필요

샘플 MDX 삭제와 Gatsby 생성물 정리는 대상 경로를 다시 알리고 승인을 받은 뒤 실행합니다.

05완료 기준

완료 판정

  • 새 URL에 제목, 설명, 날짜, 태그, 본문과 이미지 두 장이 렌더링됩니다.
  • H1 하나, H2부터 시작하는 본문, 날짜 time 요소와 한국어 대체 텍스트를 유지합니다.
  • 홈 카드, SEO 메타데이터와 sitemap이 같은 실제 글 경로를 사용합니다.
  • 샘플 URL과 예전 공개 URL은 깨끗한 빌드 결과와 sitemap에 존재하지 않습니다.
  • 전체 테스트·타입 검사·빌드·네 검증기와 세 화면 확인이 통과합니다.
  • 종성님 화면 확인과 Codex 리뷰 후 명시적 승인으로만 병합합니다.

06범위 밖 · 리스크

이번에 하지 않는 것

목차, 코드 복사, 읽기 시간, 문법 강조, 이전·다음 글, 관련 글, 예전 URL 리다이렉트, 이미지 플러그인과 배포 자동화 설명은 후속 단계입니다.

가장 큰 구현 리스크는 긴 코드와 이미지가 모바일 페이지 전체를 밀어내는 것입니다. pre와 table 내부 스크롤, 이미지 자연 크기, 페이지의 scrollWidth를 실제 브라우저에서 확인합니다.

원문을 최신 튜토리얼로 완전히 다시 쓰는 것도 범위 밖입니다. 당시 경험을 보존하면서 공식 문서와 충돌하는 안내만 고칩니다.
