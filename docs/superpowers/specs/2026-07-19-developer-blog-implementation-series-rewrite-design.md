# Developer Blog 구현 이력 기반 포스트 시리즈 재구성 설계

## 목적

현재 공개된 Gatsby 블로그 글 3편을 단순한 기능 소개에서 실제 구현 이력에 근거한 실전 제작기로 전면 재작성한다. 독자는 GitHub Pages와 Gatsby를 선택한 이유를 이해하고, 핵심 코드와 변경 지점을 따라 자신의 디자인으로 개발자 블로그를 구성할 수 있어야 한다.

입사지원서에서 글을 읽는 사람에게는 사용 기술의 나열보다 요구사항, 선택 근거, 구현 경계, 문제 해결과 검증 방식이 드러나야 한다.

## 설계 원칙

- `선택 근거 + 초보자 재현 + 실제 문제 해결`을 한 시리즈 안에 결합한다.
- GitHub Pages와 Gatsby가 항상 최선이라고 주장하지 않는다. 이 블로그의 요구사항과 제약에 맞아 선택했음을 설명한다.
- 현재 등록된 글은 사실 확인용 참고 자료로만 사용한다. 새 본문은 실제 코드, 설계 문서, 검증 기록과 basic-memory 기술 기록을 기준으로 작성한다.
- 전체 파일을 본문에 복제하지 않는다. 흐름을 이해하는 데 필요한 핵심 코드 조각만 사용한다.
- 코드 조각마다 파일 위치, 역할, 독자가 자신의 디자인에 맞게 바꿀 지점을 함께 설명한다.
- 제목에 `1편`, `2편`, `3편`을 넣지 않는다. 각 글은 검색 결과에서 독립적으로 의미가 통해야 한다.
- 시리즈 순서는 본문 도입·마무리와 이전·다음 글 연결로 표현한다.
- 추정과 확정 사실을 구분한다. 구현 당시 확인하지 않은 효과나 완료되지 않은 기능을 완료된 것처럼 쓰지 않는다.

## 시리즈 구성

시리즈는 `핵심 3편 우선 교체 + 사례 중심 2편 후속 발행`으로 구성한다.

핵심 3편은 블로그를 선택하고, 콘텐츠 시스템을 만들고, 자신의 디자인을 적용하는 전체 흐름을 완성한다. 후속 2편은 목차와 코드 블록 같은 읽기 경험, Safari와 한글 줄바꿈 같은 문제 해결 사례를 독립적으로 다룬다.

## URL과 기존 글 처리

- 핵심 3편은 확정된 새 slug로 새 공개 경로를 사용한다.
- 기존 경로는 유지하지 않는다.
- 기존 경로에서 새 경로로 리다이렉트하지 않는다.
- sitemap, canonical, 홈 목록, 이전·다음 글 링크는 새 slug를 기준으로 생성되어야 한다.
- 기존 포스트 디렉터리 제거는 파일 삭제이므로 구현 단계에서 정확한 대상 경로를 고지하고 사용자 확인을 받은 뒤 수행한다.
- 기존 글에 사용한 이미지 파일은 본문 참조 여부와 별개로 자동 삭제하지 않는다.

## 공통 글 구조

각 글은 다음 흐름을 기본 골격으로 사용한다.

1. 해결하려는 문제와 완성 결과
2. 요구사항과 제약
3. 선택한 접근과 다른 선택지의 장단점
4. 전체 구조 또는 데이터 흐름
5. 핵심 구현 코드
6. 자신의 블로그에 맞게 바꿀 지점
7. 실제로 마주친 문제와 수정 판단
8. 검증 방법과 체크리스트
9. 다음 글 또는 관련 사례 연결

초보자를 위해 새 개념은 쉬운 설명을 먼저 쓰고 기술 용어를 뒤에 붙인다. 코드보다 구조와 역할을 먼저 설명하고, 한 코드 블록에 여러 주제를 섞지 않는다.

## 핵심 코드 조각 기준

- 전체 파일 대신 하나의 역할만 보여주는 짧은 코드 조각을 사용한다.
- 코드 블록 위에 저장소 기준 파일 경로를 표시한다.
- 생략한 코드는 `...` 등으로 명시하고 복사만 하면 전체 기능이 완성된다는 인상을 주지 않는다.
- 설치 명령은 선택과 실행에 필요한 최소 범위만 싣는다.
- 저장소의 실제 구현과 다른 예제 코드를 새로 발명하지 않는다.
- 버전과 설정값은 작성 시점의 `package.json`, lockfile, Gatsby 설정에서 다시 확인한다.

## 핵심 1편 상세 설계

### 제목과 목표

`왜 GitHub Pages와 Gatsby였을까: 개발자 블로그 다시 만들기`

독자가 자신의 블로그 요구사항을 정리하고 GitHub Pages와 Gatsby의 적합성을 판단한 뒤, 유지보수 가능한 기본 골격과 배포 흐름을 이해하게 한다.

### 목차

1. **블로그를 다시 만들게 된 배경**
   - 배포 파일만 남아 수정과 확장이 어려웠던 상태를 설명한다.
2. **먼저 요구사항 정하기**
   - 무료 호스팅, 별도 서버 불필요, Git 기반 관리, React 활용, 디자인 자유도, Markdown 계열 글 관리를 기준으로 제시한다.
3. **GitHub Pages를 선택한 이유**
   - 정적 파일 배포와 기존 GitHub 저장소 연결이라는 장점을 설명한다.
   - 서버 기능이 필요한 서비스에는 맞지 않는다는 한계를 함께 밝힌다.
4. **Gatsby를 선택한 이유**
   - React 컴포넌트, 정적 사이트 생성, MDX·GraphQL 데이터 계층, 플러그인 생태계를 선택 근거로 설명한다.
5. **배포 산출물을 소스 프로젝트로 재구축하기**
   - 기존 공개 상태를 지키면서 소스 구조를 다시 만든 판단과 브랜치 경계를 보여준다.
6. **최소 프로젝트 구조 만들기**
   - 핵심 디렉터리와 설정 파일의 역할만 소개한다.
7. **처음부터 디자인 변경 지점 분리하기**
   - 색상, 폰트, 본문 너비, 사이트 정보를 독자가 바꿀 영역으로 제시한다.
8. **빌드와 GitHub Pages 배포 확인하기**
   - 로컬 실행, 정적 빌드, 배포 후 확인 항목을 체크리스트로 정리한다.
9. **다음 글 연결**
   - 기본 골격에 MDX 포스트와 GraphQL 페이지 생성을 연결한다고 안내한다.

### 핵심 메시지

Gatsby가 무조건 좋은 도구라서가 아니라, 정적 호스팅과 React 기반 UI, 파일 중심 콘텐츠 관리라는 프로젝트 요구사항에 맞아 선택했다.

## 핵심 2편 상세 설계

### 제목과 목표

`MDX와 GraphQL로 관리 가능한 Gatsby 블로그 만들기`

독자가 MDX 파일 하나를 추가했을 때 홈 목록과 상세 페이지가 자동으로 만들어지는 데이터 흐름을 이해하게 한다.

### 목차

1. **글을 코드에 직접 작성하면 생기는 문제**
   - 콘텐츠와 화면 코드가 섞일 때 생기는 관리 문제를 설명한다.
2. **포스트 저장 구조 정하기**
   - `content/posts/<slug>/index.mdx`와 글별 자산 관리 방식을 설명한다.
3. **frontmatter를 글의 계약으로 사용하기**
   - 제목, 설명, 발행일, 태그가 목록·상세·SEO에서 재사용되는 흐름을 보여준다.
4. **잘못된 글을 빌드 전에 발견하기**
   - 필수 값, 날짜 형식, slug 등 필요한 최소 검증만 적용한 이유를 설명한다.
5. **Gatsby가 MDX를 데이터로 바꾸는 과정**
   - GraphQL은 별도 외부 API 서버가 아니라 Gatsby가 빌드 중 수집한 데이터를 조회하는 내부 계층임을 풀어 쓴다.
6. **GraphQL로 포스트 목록 조회하기**
   - `allMdx`의 핵심 쿼리와 정렬·필터·필드 선택을 설명한다.
7. **홈 목록과 상세 페이지 연결하기**
   - 같은 MDX 데이터를 두 화면에서 사용하는 흐름을 보여준다.
8. **slug로 글 URL 만들기**
   - 포스트 폴더, frontmatter, 페이지 생성 코드와 URL의 관계를 설명한다.
9. **콘텐츠와 디자인을 분리하기**
   - 데이터 구조를 유지하면서 목록과 상세 UI를 자유롭게 바꿀 수 있음을 보여준다.
10. **새 글 등록 체크리스트**
    - 폴더 생성, frontmatter, 본문, 로컬 확인, 빌드 검증 순서로 마무리한다.

### 시각 자료

`MDX 파일 → Gatsby 데이터 계층 → GraphQL 조회 → 홈 목록·상세 페이지` 흐름을 작은 다이어그램으로 제시한다.

## 핵심 3편 상세 설계

### 제목과 목표

`Tailwind CSS로 내 디자인의 개발자 블로그 완성하기`

독자가 완성 예제를 복제하는 데 그치지 않고 자신의 색상, 폰트, 레이아웃 규칙을 가진 블로그로 변경하는 방법을 이해하게 한다.

### 목차

1. **화면을 만들기 전에 디자인 규칙 정하기**
   - 색상, 폰트, 본문 너비, 간격, 반응형 기준을 먼저 정한다.
2. **UI 시안을 컴포넌트로 나누기**
   - 헤더, 태그 필터, 포스트 카드, 상세 헤더, 본문으로 나눈다.
3. **Tailwind CSS에 디자인 토큰 정의하기**
   - 배경색, 강조색, 글자색, 여백을 한곳에서 관리한다.
4. **로컬 폰트 적용하기**
   - 한글 본문과 영문·코드 글꼴을 나누고 외부 폰트 요청을 만들지 않은 이유를 설명한다.
5. **824px 콘텐츠 레이아웃 통일하기**
   - 헤더, 제목, 본문의 오른쪽 끝을 상위 컨테이너의 가용 폭으로 맞춘 판단을 설명한다.
6. **홈 목록과 포스트 화면 스타일링하기**
   - 2편의 데이터 구조를 유지하면서 표현만 바꾸는 방법을 보여준다.
7. **모바일 레이아웃 설계하기**
   - 여백, 제목 크기, 태그 줄바꿈, 터치 영역을 모바일 기준으로 조정한다.
8. **Tailwind Preflight와 브라우저 기본 스타일 확인하기**
   - 제목, 목록, 링크 같은 문서 의미 스타일을 명시적으로 복구한 과정을 설명한다.
9. **내 디자인으로 바꾸는 체크포인트**
   - 색상, 폰트, 최대 너비, 구분선, 제목 크기를 표로 정리한다.
10. **여러 화면 크기에서 검증하기**
    - 데스크톱, 태블릿, 모바일의 최소 확인 항목으로 마무리한다.

### 범위 제한

목차, 코드 복사, 스크롤 상단 버튼과 Safari 전용 문제는 후속 사례 글로 분리한다.

## 후속 4편 상세 설계

### 제목과 목표

`긴 기술 글을 읽기 쉽게: Gatsby 블로그 읽기 경험 개선기`

목차, 코드 블록, 복사 버튼, 스크롤 상단 버튼을 기능 목록이 아니라 독자의 읽는 흐름을 개선한 하나의 사례로 연결한다.

### 목차

1. 긴 글에서 발견한 세 가지 불편
2. Vesper 기반 코드 강조와 언어 이름 표시
3. 복사 아이콘과 눌림 피드백의 UI 판단
4. MDX 제목을 목차 데이터와 heading id로 연결하기
5. 데스크톱 오른쪽 목차의 간격, 긴 제목 줄바꿈, 가로 넘침 조정
6. 스크롤에 따른 현재 목차 표시
7. 홈과 포스트에서 재사용하는 스크롤 상단 버튼
8. 1초 이동, 연속 동작 취소, 모션 감소와 키보드 접근
9. 여러 뷰포트, 긴 목차, 긴 코드, SSR 조건의 검증

### 핵심 메시지

기능을 많이 넣는 것보다 독자의 읽는 흐름을 방해하지 않도록 기능의 표시 시점과 피드백을 조절하는 것이 중요하다.

## 후속 5편 상세 설계

### 제목과 목표

`Chrome에서는 괜찮았는데 Safari만 좁아 보였다: Gatsby CSS 줄바꿈 디버깅`

컨테이너 폭 문제처럼 보인 현상을 브라우저 엔진별 줄 폭, 계산된 CSS, 정적 산출물로 분해해 실제 원인을 확인하는 과정을 보여준다.

### 목차

1. Chrome과 Safari에서 다르게 보인 본문
2. padding, grid, overflow를 먼저 후보로 둔 이유
3. Chromium과 WebKit에서 박스 폭과 실제 줄 폭을 따로 측정하기
4. WebKit의 `text-wrap: pretty` 줄 배치 차이 확인
5. 전역 규칙을 지우지 않고 MDX 본문에만 `text-wrap: wrap` 적용하기
6. 모바일 카드 제목의 `text-wrap: balance`와 `overflow-wrap: anywhere` 사례
7. 카드 범위에 `wrap`, `keep-all`, `break-word`를 조합한 이유
8. CSS 수정 뒤에도 이전 화면이 남은 Gatsby 증분 빌드 문제
9. clean build 후 정적 HTML, CSS hash, 브라우저 계산값 확인
10. 박스 폭부터 브라우저 엔진까지 단계별 디버깅 체크리스트

### 검증 근거

- 390px 뷰포트에서 Chromium과 WebKit의 본문 상자는 모두 342px였다.
- 수정 전 WebKit의 최대 줄 사용 폭은 약 272px였고, `text-wrap: wrap` 적용 후 약 333px로 회복됐다.
- 카드 제목은 컨테이너 폭이 아니라 전역 heading 줄바꿈 규칙과 링크 overflow 규칙의 조합이 원인이었다.
- CSS 수용 검증은 Gatsby clean production build 뒤 생성 HTML이 새 CSS hash를 참조하는 상태를 기준으로 한다.
