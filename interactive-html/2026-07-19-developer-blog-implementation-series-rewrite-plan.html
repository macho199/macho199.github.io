01배경 · 왜 지금 재구성하는가

현재 3편은 구현 요소를 설명하지만, 기술 선택 이유와 독자가 자신의 디자인으로 바꾸는 흐름이 약합니다.

새 시리즈는 GitHub Pages와 Gatsby를 선택한 요구사항부터 시작합니다. 이어서 MDX·GraphQL 콘텐츠 파이프라인과 Tailwind CSS 사용자화를 설명합니다. 목차·코드 복사와 Safari 디버깅은 핵심 흐름을 흐리지 않도록 후속 글로 분리합니다.

독자가 얻는 것

특정 완성 화면을 복제하는 방법보다 기술을 고르는 기준, 글이 URL이 되는 과정, 색상·폰트·폭을 바꾸는 위치를 이해합니다.

02접근 · 설계 결정

앱을 고치지 않고 포스트 입력과 이를 확인하는 계약만 함께 바꿉니다.

Before — 기능 중심 3편
기존 제목·slug1편·2편·3편 표기
→
기능 설명선택 이유·사용자화 약함
After — 구현 이력 기반 3편
새 제목·slug검색 결과에서 독립 의미
같은 계약→
정적 산출물홈·상세·canonical·sitemap
대안판단
기존 URL 유지링크 안정성은 높지만 새 글 구조와 slug를 맞추려는 사용자 결정을 반영하지 못해 제외합니다.
새 URL + 리다이렉트검색 이전 단계에서 불필요한 호환 계층을 만들지 않기로 결정해 제외합니다.
5편 동시 작성핵심 3편 완성이 늦어져 사례 글 2편은 후속 발행으로 분리합니다.
앱·스타일까지 변경현재 기능은 수용됐으므로 콘텐츠와 검증기만 바꾸는 최소 범위를 선택합니다.

03확정 vs 추정

상태근거와 처리
확정제목·slug·발행일, 새 URL 사용, 리다이렉트 없음, 핵심 3편 + 후속 2편 구조를 사용자에게 승인받았습니다.
확정현재 기준선은 Node 테스트 78/78, 타입 오류 0이며 애플리케이션 UI 변경은 필요하지 않습니다.
추정일반 Gatsby build 뒤 이전 포스트 디렉터리가 public/에 남을 수 있습니다. 실제 verifier 실패로 확인될 때만 승인 후 clean build합니다.
추정새 태그가 10개로 늘어 모바일에서 더 많이 줄바꿈될 수 있습니다. 기존 wrap 계약은 유지하고 브라우저 수용 검증에서 확인합니다.

05완료 기준

완료 판정

새 세 URL이 홈·상세·canonical·sitemap에 생성되고, 기존 세 URL은 없으며, 발행일과 이전·다음 글 순서가 유지되어야 합니다.

게이트통과 조건
콘텐츠승인된 제목·slug·날짜·목차이며 `1편·2편·3편` 표기가 없습니다.
소스npm test 78/78, 타입 오류 0, diff 오류 0입니다.
산출물verifier 4종이 새 메타데이터·navigation·TOC·SEO·sitemap을 확인합니다.
브라우저Chromium·WebKit에서 가로 넘침과 console 오류가 없고 긴 한글 제목·본문이 정상 줄바꿈됩니다.

06범위 밖 · 리스크

이번 범위 밖

후속 사례 2편, GA4, Search Console, 추가 SEO, About 페이지, UI·CSS·의존성 변경은 이번 계획에 포함하지 않습니다.

삭제 게이트

기존 포스트 경로 이동과 Gatsby .cache/·public/ 삭제는 실행 직전에 정확한 대상을 알리고 사용자 승인을 받아야 합니다.

새 태그가 10개로 늘어 모바일에서 줄 수가 증가할 수 있습니다. 기존 flex wrap을 유지하고 실제 390px 화면에서 수용 여부만 확인합니다. UI 재설계가 필요하면 별도 작업으로 분리합니다.
