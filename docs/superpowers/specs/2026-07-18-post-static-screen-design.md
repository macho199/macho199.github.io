# Developer Blog 포스트 정적 화면 설계

## 목적

현재 Gatsby 포스트 템플릿을 사용자가 제공한 `post.html`의 정적 읽기 화면으로 교체한다. 기존 공개 글 `Gatsby로 블로그 사이트 만들기 1편 - 시작하기`를 보수적으로 교정해 실제 MDX 콘텐츠로 이전하고, 아직 구현하지 않은 상호작용은 노출하지 않는다.

## 범위

### 포함

- 목록 복귀 내비게이션
- 태그, 제목, 설명, 발행일로 구성한 포스트 헤더
- 실제 MDX 본문의 제목·문단·강조·링크·목록·인용문·코드·이미지·구분선 표현
- 기존 공개 글의 보수적 교정과 새 슬러그 이전
- 기존 이미지 두 장의 저장소 자체 제공
- 1440px·1020px·390px 반응형 표현

### 제외

- 읽는 시간 계산과 표시
- 목차와 현재 섹션 추적
- 코드 복사와 문법 강조
- 헤딩 링크 복사
- 맨 위로 이동 버튼
- 이전·다음 글
- 시리즈와 관련 글
- 이전 URL 리다이렉트
- `updatedAt` frontmatter
- 새 런타임 또는 빌드 의존성

제외한 기능은 데이터와 동작을 함께 구현하는 후속 단계에서 추가한다. 동작하지 않는 컨트롤을 정적 모양만 먼저 렌더링하지 않는다.

## 콘텐츠 이전

### 새 포스트 식별자

- 제목: `Gatsby로 블로그 사이트 만들기 1편 - 시작하기`
- 슬러그: `gatsby-blog-1-getting-started`
- 경로: `/posts/gatsby-blog-1-getting-started/`
- 발행일: `2025-08-31`
- 설명: 원문의 의미를 유지하며 문장만 다듬는다.
- 태그: `Gatsby`, `GitHub Pages`, `React`, `Tailwind CSS`

기존 `/posts/create-a-blog-site-with-gatsby1/` 경로는 사용자의 결정에 따라 리다이렉트 없이 폐기한다. 새 버전 출시 후 기존 주소가 404가 되는 것은 의도한 동작이다.

### 샘플 포스트 교체

- `content/posts/mdx-foundation/index.mdx`를 제거한다.
- `content/posts/gatsby-blog-1-getting-started/index.mdx`를 실제 게시글로 추가한다.
- 홈과 프로덕션 검증 기준을 샘플 데이터가 아니라 새 실제 게시글로 교체한다.
- 삭제는 구현 단계에서 대상 경로를 다시 알린 뒤 수행한다.

### 편집 원칙

- 원문의 장 순서, 당시 경험, 문제 해결 흐름, 결론은 보존한다.
- 맞춤법·띄어쓰기·중복 표현과 과장된 단정만 다듬는다.
- 글 첫머리에 2025년 원문을 2026년 재구축 과정에서 재검증하고 교정했다는 안내를 둔다.
- 현재 공식 빠른 시작에 맞춰 기본 생성 명령을 `npm init gatsby`로 설명한다.
- 특정 Node.js 버전을 보편적 필수 조건처럼 단정하지 않고 프로젝트의 `engines`와 공식 지원 범위를 확인하도록 안내한다.
- `@mdx-js/react` 2.3.0 강제 다운그레이드는 현재 해결책으로 권장하지 않는다. 필요한 경우 당시 시행착오로만 설명한다.
- `npm audit fix`를 경고에 대한 일반 해결책으로 권장하지 않는다. 영향 분석과 별도 취약점 추적이 필요하다고 정정한다.
- 빌드 결과는 `file://`로 열지 않고 `gatsby serve` 또는 저장소의 `npm run serve`로 확인하도록 정정한다.
- 빌드 결과를 `main`에 직접 복사한 절차는 당시 배포 방식으로 구분한다. 현재 재구축 저장소의 자동 배포 방식은 실제 구현이 끝난 뒤 별도 글에서 다룬다.
- `Github`, `Javascript` 같은 표기를 `GitHub`, `JavaScript`로 정리한다.
- 원문을 최신 가이드로 완전히 다시 쓰거나 당시 경험을 지우지 않는다.

### 이미지

기존 글의 다음 이미지 두 장을 새 저장소에서 자체 제공한다.

- `static/images/posts/gatsby-blog-1-getting-started/hello-gatsby-tailwindcss.png`
- `static/images/posts/gatsby-blog-1-getting-started/github-pages-setting.png`

- 기존 공개 사이트의 이미지 파일을 원본으로 사용한다.
- 외부 URL을 런타임 의존성으로 남기지 않는다.
- 각 이미지에는 내용을 설명하는 한국어 대체 텍스트를 제공한다.
- 이번 단계에서는 Gatsby 이미지 변환 플러그인을 추가하지 않는다.

## 컴포넌트 경계

### 포스트 템플릿

- GraphQL 조회와 페이지 조립을 소유한다.
- 공통 `Layout`과 920px `ContentContainer`를 재사용한다.
- 목록 복귀 링크를 `<article>` 앞의 `<nav aria-label="게시글 탐색">` 안에 렌더링한다.
- `PostHeader`에 타입이 지정된 frontmatter 표현 데이터를 전달한다.
- MDX `children`을 `.mdx-content` 안에 배치한다.
- 전체 게시글을 하나의 `<article className="post-page">`로 표현한다.
- SEO 경계와 Gatsby Head API는 기존 템플릿에 유지한다.

### `PostHeader`

- 태그, 제목, 설명, 발행일만 렌더링한다.
- GraphQL이나 MDX 처리에 직접 의존하지 않는다.
- 문서의 유일한 `h1`을 소유한다.
- 화면 날짜는 `YYYY.MM.DD`, `time`의 `dateTime`은 `YYYY-MM-DD`를 사용한다.
- 태그는 현재 단계에서 상호작용 없는 정보로 표시한다.

목록 복귀 링크는 게시글 메타데이터가 아니므로 `PostHeader`에 넣지 않는다. `PostBody` 같은 단순 전달용 컴포넌트는 만들지 않는다.

## 데이터 흐름

```text
MDX frontmatter
  -> Gatsby 포스트 GraphQL 쿼리
  -> 포스트 템플릿의 타입 지정 데이터
  -> PostHeader

MDX body
  -> gatsby-plugin-mdx
  -> 포스트 템플릿 children
  -> .mdx-content
```

- 기존 필수 frontmatter인 `title`, `slug`, `publishedAt`, `description`, `tags`를 사용한다.
- GraphQL은 기계 판독용 `YYYY-MM-DD`와 화면용 `YYYY.MM.DD`를 별도 필드로 제공한다.
- 기존 frontmatter 검증은 누락되거나 잘못된 게시글을 빌드에서 거부한다.
- 새 데이터 필드와 기본값을 추가하지 않는다.

## MDX 표현 경계

- 글 본문은 의미 있는 Markdown과 MDX로 작성하고 Tailwind 클래스를 직접 넣지 않는다.
- `.post-page .mdx-content` 아래의 의미 요소를 `post.css`에서 스타일링한다.
- `gatsby-browser.js`에서 `post.css`를 한 번 불러오되 모든 포스트 규칙을 `.post-page` 또는 포스트 전용 내비게이션 클래스 아래로 제한한다.
- H1은 포스트 헤더 전용이며 MDX 본문은 H2부터 시작한다.
- fenced code block은 정적 `<pre><code>`로 렌더링하고 긴 내용만 내부에서 가로 스크롤한다.
- 코드 언어 정보는 Markdown에 남기되 이번 단계에서 토큰 단위 문법 강조는 하지 않는다.
- 표는 본문 너비보다 넓을 때 표 영역 안에서만 가로 스크롤한다.
- 이미지와 긴 URL은 페이지 전체 너비를 밀어내지 않는다.
- 외부 링크는 원문 의도를 유지하되 교정 시점의 공식 주소를 사용한다.

## 시각·반응형 규칙

### 포스트 헤더

- 왼쪽 정렬로 목록 복귀 링크, 태그, 제목, 설명, 발행일 순서를 사용한다.
- 태그는 모노스페이스 글꼴과 강조색 `#` 표식을 사용한다.
- 제목은 데스크톱에서 약 40px이며 모바일에서는 `clamp()`로 30~40px 범위에 둔다.
- 설명은 최대 760px에서 본문보다 약간 큰 도입 문장으로 표현한다.
- 발행일은 모노스페이스 글꼴과 tabular number를 사용한다.
- 헤더 아래에는 본문과 구분하는 1px 선을 둔다.

### 포스트 본문

- 920px 공통 컨테이너 안에서 최대 760px, 왼쪽 정렬을 유지한다.
- H2~H5, 문단, 강조, 링크, 중첩 목록, 인용문, 인라인 코드, 코드 블록, 이미지, 구분선에 일관된 수직 리듬을 적용한다.
- 코드 블록은 어두운 배경과 밝은 글자를 사용하되 복사 버튼이나 파일명 헤더는 렌더링하지 않는다.
- 이미지에는 테두리와 반경을 적용하고 원본 비율을 유지한다.
- 본문 링크는 색상에만 의존하지 않는 밑줄과 명확한 포커스 표현을 제공한다.

### 반응형

- 1020px 이하에서는 컨테이너 여백과 본문 상하 간격을 줄인다.
- 720px 이하에서는 제목 크기, 헤더 간격, 목록 들여쓰기, 인용문 여백을 추가로 줄인다.
- 1440px·1020px·390px에서 페이지 전체 가로 넘침이 없어야 한다.

## 접근성

- 페이지에는 포스트 제목 `h1` 하나만 둔다.
- MDX 본문은 논리적인 H2~H5 순서를 사용한다.
- 목록 복귀 링크와 본문 링크는 키보드로 접근할 수 있어야 한다.
- 태그를 버튼이나 링크처럼 표현하지 않는다.
- 발행일은 `time` 요소로 표현한다.
- 이미지 두 장 모두 의미 있는 한국어 `alt`를 제공한다.
- 표의 헤더는 `th`로 유지한다.

## 오류 처리

- 필수 frontmatter 오류는 기존 스키마와 빌드 검증에 맡기고 임의 기본값으로 숨기지 않는다.
- 이미지 파일이 없거나 생성 HTML에서 참조되지 않으면 프로덕션 검증을 실패시킨다.
- 긴 코드, 표, URL 때문에 페이지 전체에 가로 스크롤이 생기면 완료로 보지 않는다.
- 이전 주소가 생성되지 않는 것은 오류가 아니라 명시적으로 승인된 결과다.

## 검증과 완료 조건

- `PostHeader`의 타입과 태그·제목·설명·날짜 표현을 계약 테스트로 고정한다.
- 포스트 템플릿이 GraphQL과 조립만 담당하고 목록 복귀 내비게이션과 MDX 경계를 유지하는지 확인한다.
- `post.css`의 규칙이 `.post-page`와 `.mdx-content` 밖으로 새지 않는지 확인한다.
- 프로덕션 산출물에서 다음을 확인한다.
  - `/posts/gatsby-blog-1-getting-started/` 생성
  - 제목, 설명, 태그, 원본 날짜, 화면 날짜
  - 주요 H2/H3, 코드 블록, 인용문, 목록, 이미지 두 장
  - 이미지 파일과 한국어 대체 텍스트
  - `/`로 이동하는 목록 복귀 링크
  - SEO 제목, 설명, canonical, 발행일, 태그
  - TOC, 복사 버튼, 읽는 시간, 이전·다음 글, 관련 글 미노출
- 홈 검증 기준을 새 실제 게시글로 교체한다.
- `/posts/mdx-foundation/`와 `/posts/create-a-blog-site-with-gatsby1/`가 새 빌드에서 생성되지 않는지 확인한다.
- `npm test`와 `npm run typecheck`가 통과한다.
- 프로덕션 빌드와 기존 스타일·레이아웃·홈 검증이 통과한다.
- 포스트 전용 프로덕션 검증 명령이 통과한다.
- 1440px·1020px·390px에서 줄바꿈, 내부 스크롤, 이미지 비율, 포커스, 가로 넘침을 확인한다.
- 종성님 화면 확인과 Codex 리뷰 후 명시적 승인을 받아 병합한다.

픽셀 단위 스냅샷이나 CSS 문자열 전체를 고정하는 테스트는 만들지 않는다.
