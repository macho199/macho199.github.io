앞선 글에서는 `content/posts/<slug>/index.mdx` 구조와 frontmatter 계약을 정하고, 잘못된 날짜·slug·태그를 페이지 생성 전에 거부했습니다. 이제 검증을 통과한 MDX 노드를 실제 화면과 URL로 연결할 차례입니다.

이 블로그에서 GraphQL은 별도 API 서버에 요청하는 런타임 통신 계층이 아닙니다. Gatsby가 빌드하면서 수집한 데이터를 홈 페이지와 정적 페이지 생성 코드가 일관된 형태로 조회하는 내부 데이터 계층입니다.

## 같은 MDX 노드를 두 경로에서 조회하기

하나의 글은 두 경로에서 사용됩니다.

```text
Mdx 노드
  ├─ 홈 페이지: allMdx → PostList → PostCard
  └─ 상세 페이지: createPages → context.id → mdx(id) → PostHeader + 본문
```

홈은 모든 글의 요약을 발행일 역순으로 보여줘야 합니다. 상세 페이지는 특정 글 하나의 메타데이터와 MDX 본문을 렌더링해야 합니다.

두 경로가 frontmatter를 복제하지 않고 같은 `Mdx` 노드를 조회하므로 제목이나 설명을 고치면 홈, 상세, SEO가 함께 바뀝니다.

## 홈에서 발행일 역순 목록 만들기

Gatsby의 `src/pages/index.tsx`는 페이지 쿼리로 모든 MDX 글을 가져옵니다.

```graphql
query IndexPage {
  allMdx(sort: { frontmatter: { publishedAt: DESC } }) {
    nodes {
      id
      frontmatter {
        title
        slug
        publishedAt(formatString: "YYYY-MM-DD")
        publishedAtDisplay: publishedAt(formatString: "YYYY.MM.DD")
        description
        tags
      }
    }
  }
}
```

정렬은 컴포넌트에서 다시 하지 않고 GraphQL의 `sort` 인자로 요청합니다. 원본 날짜는 `time` 요소와 SEO에 사용하고, 화면 날짜는 alias를 붙인 `publishedAtDisplay`로 받습니다.

페이지는 GraphQL 노드를 UI가 사용할 `PostSummary`로 변환합니다.

```typescript
const posts: PostSummary[] = data.allMdx.nodes.map(({ id, frontmatter }) => ({
  id,
  ...frontmatter,
}))
```

`PostList`와 `PostCard`는 GraphQL을 알지 못하고 타입이 지정된 props만 받습니다. 데이터 조회와 표현을 분리하면 카드 UI를 바꿀 때 Gatsby 쿼리까지 함께 수정할 필요가 없습니다.

현재 설정에서 MDX 소스는 `posts` 하나이므로 홈의 `allMdx`가 곧 블로그 글 목록입니다. 문서나 다른 MDX 소스가 추가되면 홈 쿼리에도 명시적인 소스 필터가 필요하다는 경계는 남아 있습니다.

## 블로그 글만 정적 라우트로 생성하기

`gatsby-node.mjs`의 `createPages`도 GraphQL로 MDX 노드를 조회하지만 목적은 목록 표시가 아니라 라우트 생성입니다.

```graphql
query CreateMdxPostPages {
  allMdx {
    nodes {
      id
      internal {
        contentFilePath
      }
      parent {
        ... on File {
          sourceInstanceName
        }
      }
      frontmatter {
        title
        slug
        publishedAt
        description
        tags
      }
    }
  }
}
```

조회 결과 중 `gatsby-source-filesystem`에서 `posts`로 이름 붙인 소스만 선택합니다.

```javascript
const posts = nodes.filter(
  post => post.parent?.sourceInstanceName === "posts",
)
```

이 필터는 향후 다른 MDX 컬렉션이 생겨도 문서 페이지를 블로그 URL로 잘못 만들지 않게 합니다. 선택한 글은 `validatePostNodes`를 통과한 뒤에만 라우트 생성으로 넘어갑니다.

## id로 상세 데이터와 본문 연결하기

각 글은 frontmatter slug로 공개 URL을 만들고, `id`를 page context에 전달합니다.

```javascript
actions.createPage({
  path: `/posts/${post.frontmatter.slug}/`,
  component: `${postTemplate}?__contentFilePath=${post.internal.contentFilePath}`,
  context: {
    id: post.id,
  },
})
```

여기에는 서로 다른 식별자가 두 개 있습니다.

- `slug`는 사용자가 보는 안정적인 공개 URL을 결정합니다.
- `id`는 Gatsby 데이터 계층에서 상세 노드 하나를 다시 찾습니다.

`internal.contentFilePath`는 `gatsby-plugin-mdx`가 실제 MDX 본문을 포스트 템플릿의 `children`으로 연결하는 데 사용합니다. page context에 frontmatter 전체를 복사하지 않고 최소 식별자만 전달하기 때문에 메타데이터 원본이 둘로 나뉘지 않습니다.

상세 템플릿은 전달받은 `$id`로 정확한 노드 하나를 조회합니다.

```graphql
query PostById($id: String!) {
  mdx(id: { eq: $id }) {
    frontmatter {
      title
      slug
      publishedAt(formatString: "YYYY-MM-DD")
      publishedAtDisplay: publishedAt(formatString: "YYYY.MM.DD")
      description
      tags
    }
  }
}
```

## 표현 컴포넌트와 GraphQL 경계 분리하기

상세 템플릿은 조회한 frontmatter와 MDX 본문을 조립합니다.

```tsx
<article className="post-page">
  <PostHeader post={frontmatter} />
  <div className="mdx-content">{children}</div>
</article>
```

`PostHeader`는 태그·제목·설명·발행일만 표현합니다. GraphQL 쿼리와 Gatsby 페이지 생성 책임은 템플릿과 `gatsby-node.mjs`에 남아 있습니다.

이 경계를 사용하면 다음 변경의 영향이 분명해집니다.

- 목록에 필드가 필요하면 홈 쿼리와 `PostSummary`를 함께 바꿉니다.
- 상세 메타데이터가 필요하면 템플릿 쿼리와 `PostHeaderData`를 함께 바꿉니다.
- URL 생성 규칙은 `createPages`에서만 바꿉니다.
- 카드나 헤더의 시각 표현은 GraphQL 없이 테스트할 수 있습니다.

## SEO와 sitemap을 같은 메타데이터에 연결하기

상세 페이지의 Gatsby Head API도 조회한 frontmatter를 그대로 사용합니다.

```tsx
<Seo
  title={data.mdx.frontmatter.title}
  description={data.mdx.frontmatter.description}
  pathname={location.pathname}
  type="article"
  publishedAt={data.mdx.frontmatter.publishedAt}
  tags={[...data.mdx.frontmatter.tags]}
/>
```

이 값으로 문서 title, meta description, canonical, `article:published_time`, `article:tag`를 만듭니다. 화면과 검색 메타데이터가 같은 콘텐츠 계약을 사용하므로 서로 다른 제목이나 날짜가 노출될 가능성을 줄입니다.

`gatsby-plugin-sitemap`은 빌드에서 생성된 페이지를 기준으로 sitemap을 만듭니다. 새 글을 별도 목록에 다시 등록하지 않아도 `createPage`가 만든 `/posts/<slug>/`가 sitemap에 포함됩니다.

## 빌드 결과로 전체 흐름 검증하기

소스 테스트만 통과하면 GraphQL 쿼리와 컴포넌트 문자열은 확인할 수 있지만, 실제 정적 HTML과 URL이 만들어졌다는 보장은 부족합니다. 그래서 깨끗한 Gatsby 빌드 뒤에 생성물을 다시 읽습니다.

검증 항목은 다음과 같습니다.

1. 홈에 세 글이 `2026-06-27`, `2026-05-16`, `2026-04-18` 순서로 표시됩니다.
2. 세 `/posts/<slug>/` 경로에 상세 HTML이 생성됩니다.
3. 화면 제목·설명·날짜·태그와 SEO metadata가 frontmatter와 일치합니다.
4. 세 URL이 sitemap에 포함됩니다.
5. 폐기한 샘플 URL은 다시 생성되지 않습니다.
6. 아직 구현하지 않은 필터·페이지네이션·TOC·코드 복사 컨트롤은 노출되지 않습니다.

```shell
npm test
npm run typecheck
npm run build
npm run verify:home
npm run verify:post
```

이 검증은 GraphQL 쿼리 하나만 확인하는 것이 아니라 파일에서 정적 페이지까지 이어지는 전체 경계를 고정합니다.

## 마치며

Gatsby의 GraphQL을 사용한 핵심 이유는 문법 자체가 아니었습니다. 파일에서 읽은 콘텐츠를 홈 목록, 상세 페이지, SEO와 sitemap이 같은 계약으로 소비하게 만드는 데 있었습니다.

포스트 파일 관리와 페이지 생성 흐름이 갖춰지면서 새 글을 추가하는 작업은 반복 가능한 입력과 검증의 문제가 되었습니다. 다음 기능인 태그 필터도 이 세 글의 실제 태그 데이터를 기준으로 설계할 수 있습니다.
