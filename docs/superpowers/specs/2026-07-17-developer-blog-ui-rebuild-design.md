# Developer Blog UI 재구축 설계

## 목적

Tailwind로 작성된 반응형 HTML 산출물을 현재 Gatsby 5·MDX 기반 블로그에 단계적으로 이식한다. 각 단계는 독립적으로 구현·검증할 수 있어야 하며, 동작하지 않는 UI를 먼저 노출하지 않는다.

## 배경

- 현재 `develop`에는 Gatsby 5 기반, MDX 포스트 생성, strict 날짜 검증, SEO·sitemap이 구축되어 있다.
- 디자인 참조는 홈 `index.html`과 포스트 `post.html` 두 개다.
- 참조 산출물은 Tailwind v4 브라우저 CDN과 커스텀 CSS를 함께 사용한다.
- 실제 Gatsby 구현에서는 브라우저 CDN을 사용하지 않고 로컬 빌드로 전환한다.
- 기존 GraphQL 쿼리, 포스트 URL `/posts/<slug>/`, SEO 메타데이터와 sitemap 동작은 유지한다.

## 확정한 접근

### 선택: 기반과 기능을 최대한 분리

1. Tailwind·폰트·디자인 토큰 기반
2. 공통 레이아웃
3. 홈 정적 화면
4. 포스트 정적 화면
5. 태그 필터
6. 페이지네이션
7. TOC
8. 코드 복사
9. 이전·다음 글과 관련 글

현재 단계 하나만 GitHub 하위 이슈로 만들고 구현·검증한 뒤 다음 단계를 확정한다. 전체 하위 이슈를 미리 대량 생성하지 않는다.

### 배제한 접근

- **화면과 모든 기능을 한 번에 이식**: 변경 범위가 커서 회귀 원인과 검토 경계가 불명확해진다.
- **동작하지 않는 컨트롤을 먼저 노출**: 디자인은 빨리 비슷해지지만 접근성과 사용자 신뢰를 해친다.
- **영문·한글 글꼴을 분리**: 언어별 스타일 로직은 필요 없지만 웹폰트 수와 다운로드 비용이 불필요하게 커진다.

## 스타일 기반

### Tailwind와 PostCSS

- Tailwind v4를 Gatsby의 로컬 PostCSS 빌드에 연결한다.
- Tailwind 브라우저 CDN과 런타임 스타일 생성은 사용하지 않는다.
- 참조 산출물의 색상·타이포그래피·간격·반경·그림자·포커스·모션·컨테이너 값을 의미 기반 CSS 토큰으로 정리한다.
- Tailwind 유틸리티는 정적으로 탐지 가능한 완전한 클래스 이름을 사용한다. 런타임 문자열 조합으로 클래스를 만들지 않는다.
- 첫 UI 출시는 라이트 테마만 지원한다. 다크 테마는 후속 범위다.
- Tailwind Preflight가 MDX 제목·목록·표 같은 의미 요소를 지우지 않도록 전역 기본 스타일과 MDX 본문 스타일의 책임을 분리한다.

### 글꼴

- 제목·큰 헤딩·블로그 로고: Noto Serif KR
- 본문·메뉴·태그·메타데이터: Noto Sans KR
- 코드: 시스템 고정폭 글꼴 스택
- 한글과 영문을 구분하는 런타임 로직은 두지 않는다.
- 두 Noto 글꼴은 외부 폰트 서비스 없이 자체 제공한다. 참조 산출물에 맞춰 Noto Serif KR 400과 Noto Sans KR 400·500·600만 포함한다.
- Signifier·Söhne는 라이선스와 파일이 제공되지 않았으므로 번들하지 않는다.

### 반응형 기준

- 디자인 참조의 주요 경계인 1020px·720px·430px을 유지한다.
- 데스크톱 기준 화면은 1440px, 태블릿 기준은 1020px, 모바일 기준은 390px에서 확인한다.
- 포스트 TOC는 해당 기능 단계에서만 추가하며 데스크톱 1320px 초과에서 표시하는 참조 동작을 따른다.
- 표처럼 폭이 넓은 MDX 요소는 페이지 전체가 아니라 해당 요소 내부에서만 가로 스크롤한다.

## 컴포넌트 경계

### 공통 레이아웃 단계

- `Layout`: 페이지 공통 골격과 main 영역을 소유한다.
- `Header`: 블로그 식별자와 실제로 이동 가능한 탐색 링크만 제공한다.
- `Footer`: 참조 산출물의 공통 하단 정보를 표현한다.
- 콘텐츠 컨테이너: 페이지별 너비와 좌우 여백을 일관되게 적용한다.

### 홈 정적 화면 단계

- 홈 소개 영역
- `PostList`
- `PostCard`
- 포스트 카드의 태그는 정보로만 표시한다.
- 태그 필터와 페이지네이션을 구현하기 전에는 모든 포스트를 기존 발행일 역순으로 표시한다.
- 필터·더보기·페이지네이션 컨트롤은 관련 기능 단계까지 렌더링하지 않는다.

### 포스트 정적 화면 단계

- 포스트 뒤로가기 링크
- 포스트 헤더: 태그·제목·설명·발행일
- MDX 본문 스타일: 제목, 링크, 목록, 체크 목록, 인용문, 인라인 코드, 코드 블록, 표, 이미지 영역
- 코드 블록은 표시하되 복사 버튼은 코드 복사 단계에서 추가한다.
- TOC, 이전·다음 글, 시리즈·관련 글은 각 기능 단계까지 렌더링하지 않는다.

### 상호작용 단계

- 상호작용 요소는 실제 동작과 UI를 같은 단계에서 추가한다.
- 태그 필터, 페이지네이션, 태그 더보기, TOC 활성 상태, 코드 복사는 서로 독립된 상태와 테스트 경계를 갖는다.
- 한 기능의 상태를 다른 표현 컴포넌트에 숨겨서 공유하지 않는다.

## 데이터 흐름

- Gatsby 페이지와 포스트 템플릿만 GraphQL을 조회한다.
- 표현 컴포넌트는 GraphQL에 의존하지 않고 타입이 지정된 props를 받는다.
- 홈 페이지는 기존 포스트 쿼리 결과를 발행일 역순으로 표현 모델에 전달한다.
- 포스트 템플릿은 기존 MDX 본문과 frontmatter를 표현 모델에 전달한다.
- 필터·페이지네이션·TOC 등에 필요한 추가 데이터는 해당 기능 단계에서 쿼리와 타입을 함께 확장한다.
- 기존 SEO 컴포넌트와 Gatsby Head API 경계는 변경하지 않는다.

## 오류 처리

- 기존 MDX 스키마와 slug·날짜 검증을 유지한다. 잘못된 필수 데이터는 빌드에서 즉시 실패한다.
- 필수 컴포넌트 props는 타입으로 강제하고 임의 기본값으로 데이터 오류를 숨기지 않는다.
- 자체 제공 폰트가 실패하면 시스템 폰트로 읽을 수는 있지만, 폰트 요청 실패는 해당 단계 검증에서 실패로 처리한다.
- 작동하지 않는 버튼이나 링크는 비활성 모양으로 미리 만들지 않고 렌더링 자체를 미룬다.
- 알려진 `favicon.ico` 404는 UI 기반에 포함하지 않고 별도 자산 단계에서 해결한다.

## 검증 전략

검증 비용은 변경 범위에 비례시킨다.

### 모든 구현 단계

- `npm test`
- `npm run typecheck`
- 깨끗한 프로덕션 빌드
- 변경한 경로의 기본 동작 확인

### 관련 단계에서만 수행

- Tailwind·폰트: 외부 Tailwind·폰트 CDN 요청이 없고 자체 폰트가 정상 응답하는지 확인
- 공통 레이아웃: 1440px·1020px·390px에서 가로 넘침과 공통 영역 확인
- 홈·포스트: 참조 산출물과 주요 구조·간격·색상·타이포그래피 비교
- 상호작용: 키보드 조작, 포커스, 상태 변경, 새 콘솔 오류 확인
- 전체 UI 완료: 홈과 포스트를 대표 화면 크기에서 한 번 종합 검증

픽셀 단위 스냅샷은 글꼴 렌더링과 브라우저 차이에 지나치게 민감하므로 사용하지 않는다.

## 첫 구현 단위

첫 GitHub 하위 이슈와 구현 계획은 **Tailwind·폰트·디자인 토큰 기반**만 다룬다.

### 포함

- Gatsby 로컬 Tailwind v4·PostCSS 연결
- 라이트 테마 의미 기반 토큰
- Noto Serif KR·Noto Sans KR 자체 제공
- 전역 기본 스타일과 MDX 스타일 경계
- 기존 홈·포스트·SEO·sitemap 회귀 검증

### 제외

- `Layout`·`Header`·`Footer`
- 홈·포스트 디자인 이식
- 태그 필터·페이지네이션·더보기
- TOC·코드 복사·이전·다음 글·관련 글
- 다크 테마
- favicon과 이미지 처리

### 완료 조건

- Tailwind가 로컬 PostCSS 빌드로 생성된다.
- 외부 Tailwind·폰트 요청이 없다.
- 두 Noto 글꼴이 자체 자산으로 제공되고 지정 영역에 적용할 수 있다.
- 전역 스타일이 기존 MDX 의미 요소의 표현 책임을 침범하지 않는다.
- `npm test`, `npm run typecheck`, 깨끗한 프로덕션 빌드가 통과한다.
- 기존 홈과 `/posts/mdx-foundation/`이 정상 생성된다.

## 기록과 작업 관리

- Redmine #25는 사용자 결과와 전체 진행 상태를 관리한다.
- GitHub 대표 이슈 #1은 구현 단계의 목차를 관리한다.
- 현재 단계만 GitHub 하위 이슈로 만들고 완료 후 다음 단계를 합의한다.
- 이 문서는 정확한 구현 계약을 소유한다.
- basic-memory 기술 기록은 선택 근거와 블로그 포스트 소재만 요약하고 이 문서를 복제하지 않는다.
