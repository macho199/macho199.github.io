블로그 화면을 React로 만들었다면 다음 문제는 글을 어디에 저장하고 어떻게 화면으로 연결할지입니다. 글마다 페이지 컴포넌트를 하나씩 만들 수도 있지만, 글이 늘어날수록 목록·상세·SEO 정보를 여러 곳에서 반복하게 됩니다.

이 블로그는 글 하나를 `content/posts/<slug>/index.mdx` 파일 하나로 관리합니다. Gatsby는 빌드할 때 이 파일을 데이터로 바꾸고, GraphQL로 홈 목록과 상세 페이지에 전달합니다. 여기서 GraphQL은 외부 서버에 요청하는 통신 기술이 아니라 **Gatsby가 빌드 중 수집한 데이터를 일관된 형태로 조회하는 내부 계층**입니다.

이 글에서는 새 MDX 파일이 공개 URL이 되기까지의 흐름을 실제 코드와 함께 연결합니다.

## 글을 코드에 직접 작성하면 생기는 문제

React 컴포넌트 안에 글 제목과 본문을 직접 작성하면 처음에는 단순해 보입니다. 하지만 글을 추가할 때마다 다음 변경이 따라옵니다.

- 홈 목록에 제목, 설명, 날짜를 다시 입력합니다.
- 상세 페이지 경로를 직접 등록합니다.
- 문서 제목과 meta description을 별도로 작성합니다.
- 이전·다음 글 순서를 직접 연결합니다.
- sitemap에 들어갈 URL이 누락되지 않았는지 확인합니다.

같은 정보를 여러 파일에 적으면 어느 값이 원본인지 모호해집니다. 제목은 바뀌었는데 홈 카드만 이전 제목을 보여주거나, 상세 페이지는 존재하지만 sitemap에서 빠지는 문제가 생길 수 있습니다.

해결 방법은 글의 내용과 메타데이터를 한곳에 두고, 나머지 화면이 그 데이터를 읽게 만드는 것입니다. MDX 파일의 frontmatter를 단일 원본으로 삼으면 홈, 상세, SEO가 같은 값을 사용할 수 있습니다.

## 포스트 저장 구조 정하기

각 글은 slug 이름의 디렉터리 안에 `index.mdx`로 저장합니다.

```text
content/posts/<slug>/index.mdx
```

예를 들어 이 글의 파일은 다음 경로에 있습니다.

```text
content/posts/gatsby-mdx-graphql-post-system/index.mdx
```

디렉터리 이름과 frontmatter의 slug를 맞추면 저장소에서 글을 찾기 쉽습니다. 나중에 대표 이미지나 글 전용 파일이 필요하면 같은 디렉터리 안에 둘 수도 있습니다.

`gatsby-config.mjs`에서는 이 디렉터리를 `posts`라는 파일 소스로 등록합니다.

```javascript
{
  resolve: "gatsby-source-filesystem",
  options: {
    name: "posts",
    path: path.join(rootDirectory, "content", "posts"),
  },
}
```

`name: "posts"`는 단순한 표시 이름이 아닙니다. Gatsby가 다른 MDX 파일도 수집하게 됐을 때 블로그 글만 구분하는 경계가 됩니다.

## frontmatter를 글의 계약으로 사용하기

MDX 파일의 맨 위에는 YAML 형식의 frontmatter를 둡니다. 본문을 화면에 표시하는 방법과 별개로, 글을 찾고 정렬하고 설명하는 값입니다.

```yaml
title: "MDX와 GraphQL로 관리 가능한 Gatsby 블로그 만들기"
slug: "gatsby-mdx-graphql-post-system"
publishedAt: "2026-05-16"
description: "MDX 파일 구조와 frontmatter 계약을 정의하고 Gatsby GraphQL로 목록과 상세 페이지를 생성하는 콘텐츠 파이프라인을 설명합니다."
tags:
  - Gatsby
  - MDX
  - GraphQL
  - Validation
```

각 값은 다음 화면에서 재사용됩니다.

| 필드 | 사용 위치 |
| --- | --- |
| `title` | 홈 카드, 포스트 H1, 브라우저 문서 제목 |
| `slug` | `/posts/<slug>/` 공개 URL |
| `publishedAt` | 목록 정렬, 화면 날짜, article metadata |
| `description` | 홈 요약, 포스트 설명, meta description |
| `tags` | 홈 필터, 포스트 태그, article tag metadata |

Gatsby가 콘텐츠 상태에 따라 필드 타입을 추측하지 않도록 `gatsby-node.mjs`에서 GraphQL 스키마를 명시합니다.

```graphql
type MdxFrontmatter @dontInfer {
  title: String!
  slug: String!
  publishedAt: Date! @dateformat
  description: String!
  tags: [String!]!
}
```

`MdxFrontmatter @dontInfer`는 선언한 필드를 계약으로 사용하라는 뜻입니다. 느낌표가 붙은 값은 필수입니다. 글이 적거나 특정 태그 배열이 비어 있어도 GraphQL 타입이 우연히 달라지지 않습니다.

## 잘못된 글을 빌드 전에 발견하기

GraphQL 스키마는 필드의 형태를 고정하지만 작성 규칙을 전부 검사하지는 않습니다. 예를 들어 두 글이 같은 slug를 사용하거나, `2026-02-31`처럼 실제로 존재하지 않는 날짜를 적는 문제는 별도 검증이 필요합니다.

`src/lib/post-validation.mjs`의 `validatePostNodes`는 페이지를 만들기 전에 모든 글을 검사합니다.

```javascript
const requiredStringFields = ["title", "slug", "publishedAt", "description"]

for (const field of requiredStringFields) {
  const value = frontmatter[field]

  if (typeof value !== "string" || value.trim() === "") {
    errors.push(`Post ${post.id} requires a non-empty ${field}.`)
  }
}
```

검증기는 다음 규칙을 확인합니다.

- 필수 문자열이 비어 있지 않아야 합니다.
- tags는 빈 값이 없는 문자열 배열이어야 합니다.
- slug는 소문자 영문, 숫자, 단일 하이픈만 사용해야 합니다.
- 서로 다른 글이 같은 slug를 사용할 수 없습니다.
- publishedAt은 `YYYY-MM-DD` 형식의 실제 달력 날짜여야 합니다.

오류를 하나 발견했다고 즉시 멈추지 않고 전체 글의 오류를 모읍니다. `gatsby-node.mjs`는 검증 결과가 있으면 `reporter.panicOnBuild`로 빌드를 중단합니다. 덕분에 누락된 제목과 잘못된 날짜를 한 번에 고칠 수 있고, 일부 글이 조용히 빠진 채 배포되는 상황도 막습니다.

## Gatsby가 MDX를 데이터로 바꾸는 과정

파일에서 화면까지의 전체 흐름은 다음과 같습니다.

```text
content/posts/<slug>/index.mdx
  → gatsby-source-filesystem
  → gatsby-plugin-mdx
  → Gatsby Mdx 데이터
  → allMdx 홈 목록 / createPages 상세 경로
```

`gatsby-source-filesystem`은 파일을 Gatsby의 `File` 노드로 만듭니다. `gatsby-plugin-mdx`는 MDX 본문과 frontmatter를 `Mdx` 노드로 변환합니다. 이후 홈 쿼리와 `createPages`가 같은 `Mdx` 데이터를 각자의 목적에 맞게 사용합니다.

`gatsby-node.mjs`는 수집된 노드 중 부모 파일의 소스가 posts인 것만 남깁니다.

```javascript
const posts = nodes.filter(
  post => post.parent?.sourceInstanceName === "posts",
)
```

이 조건 덕분에 앞으로 문서나 다른 MDX 소스를 추가해도 모두 블로그 글 경로로 만들어지지 않습니다.

## GraphQL로 포스트 목록 조회하기

홈 페이지는 `src/pages/index.tsx`에서 모든 MDX 글을 발행일 내림차순으로 조회합니다.

```graphql
allMdx(sort: { frontmatter: { publishedAt: DESC } }) {
  nodes {
    id
    frontmatter {
      title
      slug
      publishedAt(formatString: "YYYY-MM-DD")
      publishedAtDisplay: publishedAt(formatString: "YYYY.MM.DD")
      description
      tags
    }
  }
}
```

정렬을 화면 컴포넌트에서 다시 구현하지 않고 쿼리에 선언합니다. `publishedAt`은 SEO와 데이터 비교에 사용할 원본 형식이고, `publishedAtDisplay`는 화면에 표시할 형식입니다. 같은 날짜를 두 필드에 따로 저장하지 않고 Gatsby 날짜 포맷 기능으로 만듭니다.

쿼리 결과는 `PostList`에 전달됩니다. 태그 필터와 카드 개수 표시는 이미 정렬된 글 배열을 사용하므로 콘텐츠 파일을 추가하면 홈 목록도 함께 갱신됩니다.

## 홈 목록과 상세 페이지 연결하기

홈과 상세는 같은 frontmatter를 읽지만 페이지를 만드는 방식이 다릅니다.

- 홈은 `src/pages/index.tsx` 파일 자체가 `/` 페이지가 되고 `allMdx`로 여러 글을 조회합니다.
- 상세는 `gatsby-node.mjs`가 각 글의 slug로 경로를 만들고 `src/templates/post.tsx` 템플릿을 사용합니다.

상세 템플릿은 페이지 context로 받은 id를 이용해 글 하나를 조회합니다.

```graphql
query PostById($id: String!) {
  mdx(id: { eq: $id }) {
    tableOfContents(maxDepth: 2)
    frontmatter {
      title
      slug
      publishedAt(formatString: "YYYY-MM-DD")
      publishedAtDisplay: publishedAt(formatString: "YYYY.MM.DD")
      description
      tags
    }
  }
}
```

`mdx(id: { eq: $id })`로 메타데이터를 읽고, MDX 본문은 Gatsby가 템플릿의 `children`으로 전달합니다. 템플릿은 제목·태그·목차·본문·이전·다음 글을 조립하지만 특정 글의 값을 직접 알 필요가 없습니다.

## slug로 글 URL 만들기

검증을 통과한 글은 발행일 오름차순으로 정렬한 뒤 정적 페이지가 됩니다.

```javascript
actions.createPage({
  path: `/posts/${post.frontmatter.slug}/`,
  component: `${postTemplate}?__contentFilePath=${post.internal.contentFilePath}`,
  context: {
    id: post.id,
    previousPost: toAdjacentPost(sortedPosts[index - 1]),
    nextPost: toAdjacentPost(sortedPosts[index + 1]),
  },
})
```

slug `gatsby-mdx-graphql-post-system`은 `/posts/gatsby-mdx-graphql-post-system/` URL이 됩니다. 같은 정렬 결과에서 앞뒤 글을 가져오기 때문에 상세 화면의 이전·다음 글 관계도 발행일과 일치합니다.

slug를 바꾸면 공개 URL도 바뀝니다. 따라서 단순한 파일 이름이 아니라 외부 링크와 sitemap에 영향을 주는 공개 계약으로 다뤄야 합니다.

## 콘텐츠와 디자인을 분리하기

MDX 파일은 제목, 설명, 태그, 본문처럼 글이 말하려는 내용을 가집니다. React와 CSS는 그 내용을 어떻게 보여줄지 결정합니다.

이 경계를 유지하면 다음 작업이 단순해집니다.

- 새 글을 등록할 때 홈이나 상세 컴포넌트를 수정하지 않습니다.
- 카드 디자인을 바꿀 때 모든 MDX 파일을 수정하지 않습니다.
- SEO는 frontmatter의 같은 title과 description을 사용합니다.
- 글 구조 검증과 화면 스타일 검증을 서로 다른 테스트에서 다룹니다.

MDX에서는 표와 코드 블록 같은 문서 표현을 사용할 수 있지만, 공통 UI를 글마다 복제하지 않는 편이 좋습니다. 코드 복사 버튼이나 목차처럼 모든 글에 필요한 기능은 상세 템플릿과 공통 컴포넌트에 둡니다.

## 새 글 등록 체크리스트

새 글을 추가할 때는 다음 순서로 확인합니다.

1. `content/posts/<slug>/index.mdx` 디렉터리와 파일을 만듭니다.
2. title, slug, publishedAt, description, tags를 채웁니다.
3. slug는 공개 URL로 사용할 소문자 하이픈 형식인지 확인합니다.
4. publishedAt은 실제 날짜인 `YYYY-MM-DD`로 작성합니다.
5. 본문은 H2를 기준으로 읽기 쉬운 목차를 구성합니다.
6. `npm test`와 `npm run typecheck`로 소스 계약을 확인합니다.
7. `npm run build` 뒤 홈·상세 검증기로 실제 산출물을 확인합니다.

```shell
npm test
npm run typecheck
npm run build
npm run verify:home
npm run verify:post
```

이 흐름을 사용하면 글 하나가 파일에서 데이터로, 데이터에서 목록과 상세 URL로 이어집니다. 다음 글에서는 이 데이터 구조를 그대로 둔 채 Tailwind CSS 디자인 토큰, 로컬 폰트, 824px 콘텐츠 폭을 바꾸며 자신만의 화면을 만드는 방법을 살펴봅니다.
